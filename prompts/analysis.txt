RULE 1 — GOAL GROUPING (most important rule)
═══════════════════════════════════════════

Group work into BUSINESS GOALS. A goal = the high-level outcome accomplished, not the technical steps.

IRON RULE: Everything in the same session that touches the same system is ONE GOAL.
Iterating, refining, fixing bugs, adding features — all tasks within the original goal, not separate goals.

DEFAULT RULE: When in doubt, MERGE. Too few goals is better than too many.

The only valid reason to create a new goal is: work on a COMPLETELY DIFFERENT subject in a DIFFERENT
project/session with ZERO shared files or dependencies.

ROOT/HOME FOLDER EXCEPTION: If a session's PROJECT is a user home directory (e.g. "jsmith",
"username", or a path like /Users/name or C:\Users\name), the user was likely doing ad-hoc work
that does NOT belong to a single project. In this case:
- Do NOT merge all sessions from the same home folder into one giant goal.
- Only group sessions together if they share CLEAR evidence of the same project: same files
  modified, same topic/domain, same repository, or explicit continuation of prior work.
- Separate ad-hoc tasks (e.g. a quick research question, installing a tool, answering a question)
  should each be their own small goal or grouped only with closely related tasks.
- Look at the actual CONTENT and FILES in each session, not just the project path.

GOOD GOAL TITLES (business outcome, verb-first, INCLUDE the project/repo/tool name):
  "Built a daily Copilot activity analytics tool with branded HTML reports"
  "Shipped the What-I-Did-Copilot reporting tool from concept to working system"
  "Provided strategic rewrite recommendations for the DPG-AI-Skilling presentation"
  "Diagnosed and resolved checkout regression in the Contoso API"

BAD GOAL TITLES (too granular, or missing project context):
  "Set up authentication"        ← part of a larger goal
  "Built report generator"       ← which report generator? Name the project
  "Updated README"               ← which README? Include the repo/project name
  "Initialized git repository"   ← setup step, not the goal itself
  "Improved documentation"       ← too vague — documentation for what?

TITLE RULE: The goal title must describe the PRIMARY DELIVERABLE, not the first thing done.
If a session starts with "prepare for github" but spends 80% of time building a report tool,
the title should be about the report tool, not the git setup.

ACTIVITY-TYPE RULE: Name the goal after the DOMINANT KIND OF WORK, not just the surface artifact.
Decide what most of the time was spent on, then frame the title accordingly:
  - If most of the time was RESEARCH / INVESTIGATION / ANALYSIS (reading docs, exploring code,
    comparing options, gathering evidence) — name it "Research on <subject>" or
    "Investigated <question> for <project>", not after the file that happened to get edited.
  - If most of the time was DESIGN / PLANNING / DECISION-MAKING — name it
    "Designed <thing>" or "Defined approach for <thing>".
  - If most of the time was BUILDING / SHIPPING a concrete artifact — name it after the artifact.
  - If most of the time was CLEANUP / REFACTOR / MAINTENANCE — name it that, but only if real
    code was actually moved or removed; "directory cleanup" is NOT a valid label for a session
    that was mostly research.

MANDATORY SELF-CHECK before you write a goal title:
  1. Look at the tags on this goal's tasks. Count how many tasks carry tags like
     "Technical Research", "Requirements Analysis", "Data Analysis", "Product Planning",
     "System Architecture" vs how many produced a concrete shipped artifact.
  2. If RESEARCH/ANALYSIS/PLANNING tags dominate (more than half the tasks, or more than half
     the hours), the title MUST lead with "Research on", "Investigated", "Analyzed",
     "Evaluated", "Defined", or "Designed" — NOT with "Built", "Cleaned up", "Organized",
     "Updated", "Reformatted", or any artifact-centric verb.
  3. Lame surface-action verbs like "Organized", "Cleaned up", "Reformatted", "Touched up"
     are only allowed when the work genuinely was that — i.e., real files moved/deleted/
     restructured with little reading or thinking involved.

Examples:
  ✗ "Daybreak directory cleanup"        (when 80% of the work was reading + analysis)
  ✓ "Research on Daybreak repo structure"
  ✗ "Organized Daybreak project directory and implemented publish pipeline"
                                        (when tasks are dominated by Technical Research +
                                         Requirements Analysis + Data Analysis tags)
  ✓ "Researched Daybreak repo structure and methodology for publish pipeline"
  ✓ "Investigated time-saving estimate methodology for Daybreak"
  ✗ "Updated whatidid README"           (when the real work was rethinking the install story)
  ✓ "Rewrote whatidid install guide to lead with the CLI plugin"
The "what got built" rule still applies for build sessions — but research, planning, and
decision-making are first-class deliverables and must be labeled honestly as such.

CANONICAL NAME RULE (overrides everything below): Each session header includes a line
"CANONICAL PROJECT NAME (anchor the goal label on this): <name>". That <name> is the real
workspace folder / git repository the work happened in. You MUST build the goal label around
that canonical name. NEVER name a goal after the wording of the first user message or the
chat session's auto-generated title — those reflect how the conversation opened, not which
project it was for. If several sessions share the same canonical name, they are the same
project: merge them and use one label built on that name.
  CANONICAL = "what-i-did-copilot"  →  label "What-I-Did-Copilot reporting tool" ✓
  First prompt was "fix the token count"  →  do NOT label the goal "Fix the token count" ✗
EXCEPTION: when the canonical name is a user home directory (see ROOT/HOME FOLDER EXCEPTION
above), fall back to the actual subject/files of the work to name the project.

LABEL RULE: The label must include enough context to identify the project. Use the repo name,
the primary file/tool being worked on, or the product name. Someone reading just the labels
should know WHICH project each line refers to.
  GOOD: "What-I-Did-Copilot reporting tool", "Frontier Firm report", "DPG-AI-Skilling deck"
  BAD:  "Report improvements", "Updated README", "Code refactor", "Documentation"

RULE 2 — LANGUAGE
═══════════════════════════════════════════

Write as if briefing a senior executive on what was accomplished.

NEVER write anything that implies the human was unclear, imprecise, or needed to course-correct.
NEVER ASSUME CONTEXT NOT IN THE TRANSCRIPT.
DO use the actual file or document name when it appears in the tool calls.

SPECIFICITY IS THE WHOLE POINT. Every title, label, summary, and outcome must name the CONCRETE
thing that was built, changed, or fixed — the actual feature, file, capability, or behavior. Lead
with a strong past-tense verb (Built, Added, Fixed, Wired, Replaced, Rewrote, Shipped, Integrated).
A reader who knows nothing about the project should learn exactly WHAT now exists or works.

BANNED — vague filler that says nothing. Never use these words/phrases as the payload of a sentence:
  ✗ "enhanced functionality", "improved capabilities", "better performance", "more comprehensive"
  ✗ "various updates", "several improvements", "general enhancements", "streamlined the process"
  ✗ "made changes to", "worked on", "updated X for Y" where Y is a vague benefit
  If you catch yourself writing one of these, replace it with the specific feature or behavior.

GOOD framing (concrete — names the actual thing):
  ✓ "Added multi-recipient support to the report email draft, splitting on commas and semicolons"
  ✓ "Fixed the M365 usage processor so audit-log bundles explode into per-user rows"
  ✓ "Replaced the COM send path with a prefilled .eml draft that opens in the user's mail app"

BAD framing (vague — could describe any work on any project):
  ✗ "Enhanced the PAX App with new functionality"
  ✗ "The tool now generates more comprehensive reports"
  ✗ "Made improvements to the Python script"

RULE 3 — EFFORT ESTIMATES (research-calibrated)
═══════════════════════════════════════════

human_hours = what a skilled professional would need WITHOUT AI assistance.

RESEARCH FOUNDATION:
- LLM-assisted work completes 1.4–4× faster than unassisted work
  (Cambon et al. 2023, Peng et al. 2023)
- The human EQUIVALENT is what matters: reverse the speedup to get what
  it would take without AI
- Four signals independently explain AI-assisted effort (OLS, 48 days):
    1. Conversation turns — each substantive turn = expert thinking
    2. Logic code lines — .py/.ts/.go/.rs/.java/.sh only; HTML/CSS/JSON/MD excluded
    3. Read/search calls — file reads + grep/glob/search; captures investigation
    4. Tool invocations — total tool calls; captures non-coding work (image analysis, synthesis, browser)
       The request counter (legacy PRU, now superseded by AI Credits) serves as
       fallback interaction signal when conversation turn data is unavailable.
- Boilerplate lines (HTML/CSS/JSON/MD/YAML) correlate NEGATIVELY with effort
  (r=−0.14); logic lines r=+0.41

THE FORMULA IS A FLOOR, NOT A CEILING.
The deterministic formula (base × complexity_mult, where base = interaction_h + lines_h + reads_h + tools_h)
gives the minimum plausible estimate based on mechanical counts. Your semantic judgment about
QUALITY — what decisions were made, what expertise was required, what would
an expert firm bill — should land AT OR ABOVE the formula floor, not below.
The formula cannot see business value; you can.

STEP 1 — Classify task type:
  Mechanical (install, deploy, git push, config)       → 0.25–0.5h max, always
  Research/Analysis (read-heavy, low edits)             → active_minutes × 2–3 anchor
  Implementation (edit-heavy, logic lines added)        → logic lines ÷ 100 LoC/hr + overhead
  Debugging (run-heavy, high iteration)                 → base + 50% rework premium
  Complex multi-step (balanced)                         → highest signal as floor, judgment upward

STEP 2 — Estimate from multiple signals; use the HIGHEST as your base:

  ACTIVE TIME (primary real-world anchor):
    active_minutes × 2 = conservative (routine/mechanical work, 1.4× lower bound)
    active_minutes × 3 = standard (implementation, feature building)
    active_minutes × 4 = generous (design, debugging, research, decision-making)
    active_minutes × 5–6 = complex (high iteration depth + broad file scope,
      e.g. debugging across 10+ files with repeated rework, architectural refactors)
    Example: 90 min active on feature design → 3–4h human equivalent.
    Example: 60 min active on multi-file debugging with 5+ rework cycles → 5–6h.

  CONVERSATION TURNS (substantive only — exclude 'yes'/'commit'/'ok'/confirmations):
    1–3 = 0.25h  |  4–8 = 0.75h  |  9–15 = 1.5h  |  16–30 = 2.5–3h
    31–60 = 3–5h  |  61–100 = 5–8h  |  100+ = 8–12h
    Heavy iteration on the same problem → estimate toward the upper end of range.

  LOGIC CODE LINES (logic files ONLY — not boilerplate):
    INCLUDE: .py .js .ts .jsx .tsx .go .rs .java .cs .cpp .c .sh .rb .sql .kt .swift
    EXCLUDE: .html .css .json .md .yaml .yml .csv .xml .toml .lock
    Expert writes 80–130 LoC/hr of logic code.
    100 lines ≈ 1h  |  300 lines ≈ 2.5h  |  800 lines ≈ 6h

  READ/SEARCH CALLS (investigation and analysis):
    Heavy read sessions (40+ reads/searches) add 0.5–1h for navigation and judgment
    even when no code was written.

  TOOL INVOCATIONS (non-coding work signal):
    Total tool calls capture work that doesn't produce code: image analysis,
    document synthesis, browser automation, data exploration, presentation creation.
    High tool counts with low lines → the work was non-coding but still substantial.
    50 tools ≈ +0.4h  |  100 tools ≈ +0.5h  |  200+ tools ≈ +0.5–0.6h

STEP 3 — Apply COMPLEXITY MULTIPLIER based on iteration depth and file scope:
  The deterministic formula applies a bounded complexity multiplier (1.0–1.60×)
  when base effort ≥ 0.50h. Your AI estimate should reflect the same logic:

  ITERATION DEPTH (avg edits per file — measures rework/debugging intensity):
    ≥ 2.5 edits/file → +10%    (moderate rework)
    ≥ 5.0 edits/file → +25%    (heavy debugging/iteration cycles)
    ≥ 10  edits/file → +35%    (extreme rework — multiple failed approaches)

  FILE SCOPE (unique files touched — measures breadth and context-switching):
    ≥ 5 files  → +10%     (multi-file change)
    ≥ 10 files → +25%     (broad architectural change)

  These stack: a session with iteration_depth=8 and 12 files touched gets up to
  +50% (capped at 1.60×). This reflects Tregubov et al. (2017) finding that 17%
  of developer time is spent recovering from context switches.

STEP 4 — Apply additional upward semantic adjustments based on transcript content:
  Significant rework or debugging cycles beyond what iteration_depth captures: +10–20% on base
  Architecturally significant decisions (APIs, schemas):     scale to upper anchors
  Novel problem-solving (creative synthesis, not routine):   scale to upper anchors
  Pure mechanical execution only:                            cap at 0.25–0.5h

STEP 5 — Reality-check against outcome anchors:
  0.25h    — Trivial: install package, run CLI command, toggle config, push to git
  0.5h     — Simple: minor code edit, rename, format tweak, answer a question
  1.0–1.5h — Moderate: implement small feature, debug known issue, draft short doc
  2–4h     — Substantial: design + implement feature, complex analysis, detailed report
  4–8h     — Major: architect new module, build complete tool, comprehensive research
  8–16h    — Large: full system from scratch, multi-day design-implement-test cycle

CAPS AND FLOORS:
- Mechanical execution (install, deploy, git ops, copy files) → 0.25–0.5h max always
- No single task should exceed 8h — split into sub-tasks if that large
- Each number must be nearest 0.25h
- goal.human_hours must exactly equal the sum of its task hours
